# Hardware Description Document

## 1. Purpose and scope

This document describes the YJSP Recovery Board from a software-facing
hardware perspective.

It covers:

- board-level functional architecture
- software-visible subsystems and interfaces
- interconnects between those subsystems
- relevant connector mappings
- control, telemetry, safety, and fault paths
- nominal hardware behavior suitable for firmware understanding and
  Renode-style logical modeling

It does **not** attempt to be a full electrical design review. Passive
component values, layout details, manufacturing notes, and non-nominal
assembly variants are out of scope except where they affect observable
behavior.

This document describes the **nominal** board configuration only.

## 2. Board overview

At a high level, the board consists of five major functional domains:

1. **Power input and conversion**
   - two independent 48 V battery inputs
   - two independent 48 V to 24 V converters
   - active 24 V power OR
   - two independent 24 V to 3.3 V converters
   - active 3.3 V power OR

2. **Three redundant flight-control microcontrollers**
   - MCU A
   - MCU B
   - MCU C

3. **Three independent sensor suites**
   - one dedicated sensor suite per MCU
   - each suite contains an IMU, magnetometer, and barometer

4. **Stage voting logic**
   - stage 1 majority vote
   - stage 2 majority vote

5. **Recovery output drivers**
   - five driver blocks: A, B, C, D, E
   - each driver block contains two output channels
   - each channel exports a `PWR` node and an `RTN` node

The board exposes a shared SPI-style interface to the external flight
computer, plus separate safety inputs for RBF and E-stop.

## 3. High-level nominal architecture

### 3.1 Power tree

The nominal power tree is:

- `VIN1` and `VIN2` are two independent 48 V inputs.
- `VIN1` feeds one 48 V to 24 V buck converter.
- `VIN2` feeds a second 48 V to 24 V buck converter.
- The two 24 V outputs are ORed into the board-wide `P24V0` rail.
- `P24V0` feeds two independent 24 V to 3.3 V buck converters.
- The two 3.3 V outputs are ORed into the board-wide `P3V3` rail.
- `P24V0` also feeds each recovery driver block, which locally derives a
  `+12V_RECO_<X>` rail for that driver's analog and diagnostic circuitry.

### 3.2 Compute and sensing

There are three identical STM32H573-based controller instances:

- **MCU A**
- **MCU B**
- **MCU C**

Each MCU has its own dedicated sensor suite connected on a local SPI
bus:

- one IMU
- one magnetometer
- one barometer / thermometer

The sensor suites are not shared between MCUs.

### 3.3 Redundant actuation decision path

The board implements a deliberate fault-tolerant actuation architecture.

Each stage enable is generated by a **2-of-3 majority vote** across the
three MCUs:

- **Stage 1 voted enable** is produced from
  `RECO1A_EN`, `RECO1B_EN`, and `RECO1C_EN`.
- **Stage 2 voted enable** is produced from
  `RECO2A_EN`, `RECO2B_EN`, and `RECO2C_EN`.

The voting logic is explicitly documented in the schematic as:

`Output = AB + BC + AC`

where:

- `A` is MCU A's vote
- `B` is MCU B's vote
- `C` is MCU C's vote

This means a single MCU cannot command a stage by itself.

### 3.4 Recovery driver grouping

The five driver blocks are partitioned by stage as follows:

- **Stage 1 driver group**: drivers **A**, **B**, and **C**
- **Stage 2 driver group**: drivers **D** and **E**

Each driver block has two channels named `RECO1` and `RECO2`.

In the nominal top-level wiring:

- for drivers **A**, **B**, and **C**, both `RECO1_EN` and `RECO2_EN`
  are tied to the **stage 1 voted enable**
- for drivers **D** and **E**, both `RECO1_EN` and `RECO2_EN` are tied
  to the **stage 2 voted enable**

As a result, when a stage voted enable asserts, **both channels inside
that stage's driver blocks are enabled together**.

## 4. External interfaces

## 4.1 Battery / power inputs

### 4.1.1 `VIN1` input connector (`J702`)

Nominal function:

- provides one 48 V battery input path to the board
- feeds the first 48 V to 24 V buck stage

### 4.1.2 `VIN2` input connector (`J704`)

Nominal function:

- provides the second 48 V battery input path to the board
- feeds the second 48 V to 24 V buck stage

## 4.2 Flight computer interface (`J703`)

The external flight computer communicates with the recovery board over a
shared SPI-style bus.

The recovery board presents **three independent SPI slave endpoints** to
the flight computer, one per MCU. The clock and data lines are shared;
chip-select is separate per MCU.

### 4.2.1 Logical interface

Shared signals:

- `uC_SCLK`
- `uC_MOSI`
- `uC_MISO`

Per-MCU chip-selects:

- `nCS-uC-A`
- `nCS-uC-B`
- `nCS-uC-C`

Interpretation:

- the **flight computer is the bus master**
- each STM32 is a separate slave device on the shared bus
- only the MCU whose chip-select is asserted should respond

### 4.2.2 Connector pin mapping for `J703`

| Pin | Signal |
|---|---|
| 1 | `SCLK-uC-R` |
| 2 | `GND` |
| 3 | `MOSI-uC-R` |
| 4 | `GND` |
| 5 | `MISO-uC-R` |
| 6 | `nCS-uC-B` |
| 7 | `GND` |
| 8 | `nCS-uC-C` |
| 9 | `GND` |
| 10 | `nCS-uC-A` |

Notes:

- the schematic includes series elements on the shared bus lines for
  harness-transient mitigation
- these do not change the logical interface model

## 4.3 Remove-before-flight tag (`J701`)

The RBF interface provides a **hardware safing input** to the recovery
board.

Nominal behavior:

- the net is named `RBFTAG_DISABLE`
- the schematic explicitly notes that **when plugged in, RBF = 3V3**
- local pulldown bias holds the signal low when not driven high

Software-visible meaning:

- when `RBFTAG_DISABLE` is asserted, recovery driver firing is hardware
  inhibited
- this is a hardware safety path that bypasses software intent

## 4.4 E-stop input (`J705`)

The E-stop interface provides a second hardware inhibit path.

Nominal behavior:

- the net is named `ESTOP_DISABLE`
- the signal is routed to every recovery driver block
- local biasing holds the signal low unless externally driven

Software-visible meaning:

- when `ESTOP_DISABLE` is asserted, recovery driver firing is hardware
  inhibited
- this is independent of MCU-generated stage enables

The schematic does not fully specify the external E-stop source device,
so this document treats `ESTOP_DISABLE` as an externally asserted logic
input whose asserted state disables firing.

## 5. Power subsystem

## 5.1 48 V to 24 V conversion

There are two nominally identical 48 V to 24 V buck converter channels.
Each is implemented with a FAN65008B controller.

Nominal behavior:

- input rail: `P48V0`
- output rail: per-channel 24 V rail
- nominal output: 24 V
- schematic note: **48V to 24V @ 8A**
- enable/UVLO threshold is set so the channel enables when the 48 V input
  is above approximately 11 V

The two 24 V converter outputs are named `P24V0A` and `P24V0B` before ORing.

## 5.2 24 V power OR

The two independent 24 V rails are combined by an active OR stage using
RT1987N-A devices.

Nominal behavior:

- inputs: `P24V0A`, `P24V0B`
- output: `P24V0`
- channel enable thresholds are derived from the local 24 V rails on the
  main power sheet
- the OR stage should be modeled as allowing either valid input rail to
  sustain `P24V0`

There is no MCU control path into this OR stage.

## 5.3 24 V to 3.3 V conversion

There are two nominally identical 24 V to 3.3 V buck converter channels.
Each is implemented with a MAX17503.

Nominal behavior:

- input rail: `P24V0`
- outputs: `P3V3A`, `P3V3B`
- nominal output: 3.3 V
- schematic note: EN is tied to VIN for always-on operation

There is no software control of these converter enables in the nominal
design.

## 5.4 3.3 V power OR

The two 3.3 V rails are combined by an active OR stage using LM73100
power-path devices.

Nominal behavior:

- inputs: `P3V3A`, `P3V3B`
- output: `P3V3`
- UVLO and OVLO thresholds are set locally in hardware
- the OR stage should be modeled as supplying `P3V3` from either valid
  3.3 V source

There is no MCU control path into this OR stage.

## 5.5 Software-visible power telemetry

All three MCUs receive analog telemetry for the main board rails:

- `24V-VSNS`
- `3V3-VSNS`

These are generated by resistor dividers from the main rails.

Nominal interpretation:

- `24V-VSNS` is an ADC-safe scaled image of `P24V0`
- `3V3-VSNS` is an ADC-safe scaled image of `P3V3`

These signals are intended for rail monitoring, not control.

## 6. MCU subsystem

## 6.1 Controllers

The board contains three MCU instances, all based on `STM32H573RIT6`:

- MCU A
- MCU B
- MCU C

Each MCU instance has:

- its own local sensor SPI bus and dedicated sensor suite
- its own chip-select on the flight-computer SPI bus
- access to the common stage-voting control scheme
- access to the shared fault and latch infrastructure

## 6.2 Redundancy model

The controller architecture is intentionally redundant.

### 6.2.1 Per-stage command voting

Each MCU produces one vote per stage:

- stage 1 votes:
  - `RECO1A_EN`
  - `RECO1B_EN`
  - `RECO1C_EN`
- stage 2 votes:
  - `RECO2A_EN`
  - `RECO2B_EN`
  - `RECO2C_EN`

These feed discrete majority-vote hardware.

### 6.2.2 Shared fault visibility

Every MCU receives **all** recovery-driver fault outputs:

- `FLTA1`, `FLTA2`
- `FLTB1`, `FLTB2`
- `FLTC1`, `FLTC2`
- `FLTD1`, `FLTD2`
- `FLTE1`, `FLTE2`

This means any MCU can observe any driver's digital fault status.

### 6.2.3 Shared latch control

All five latch lines are shared across the MCU set:

- `LATCH_A`
- `LATCH_B`
- `LATCH_C`
- `LATCH_D`
- `LATCH_E`

These nets are tied together across the MCU sheet instances and pulled up
on the primary sheet. The schematic explicitly notes that the MCU latch
pins **must be configured as open-drain**.

Implication:

- any MCU may assert a latch-reset pulse for a given driver
- because the lines are shared, no MCU may drive them push-pull high

## 7. Sensor subsystem

Each MCU has one dedicated local sensor suite connected by a private SPI
bus.

Each suite contains:

- **IMU**: `ASM330LHBG1`
- **Magnetometer**: `LIS2MDLTR`
- **Barometer / thermometer**: `MS561101BA03-50`

## 7.1 Local sensor interconnect per MCU

Per sensor suite, the MCU provides:

- shared local SPI clock: `SCLK`
- shared local SPI MOSI: `MOSI`
- shared local SPI MISO: `MISO`
- individual chip-selects:
  - `IMU_CS`
  - `MAG_CS`
  - `BAR_CS`

Additional signal:

- `IMU_INT` from the IMU to the MCU

## 7.2 Sensor power

All sensors are powered from `P3V3`.

There is no MCU-controlled sensor power switch in the nominal design.
The sensors should be modeled as present whenever `P3V3` is present.

## 7.3 Sensor behavior summary

### 7.3.1 IMU

- device: `ASM330LHBG1`
- interface: SPI
- interrupt line: `IMU_INT`
- one device per MCU

### 7.3.2 Magnetometer

- device: `LIS2MDLTR`
- interface: SPI
- one device per MCU

### 7.3.3 Barometer / thermometer

- device: `MS561101BA03-50`
- interface: SPI
- the schematic explicitly notes that forcing `PS` low selects SPI mode
- one device per MCU

## 8. Stage voting subsystem

There are two voting-logic blocks, both using the same majority function:

- `VotingA_Block` for stage 1
- `VotingB_Block` for stage 2

## 8.1 Stage 1 vote

Inputs:

- `RECO1A_EN`
- `RECO1B_EN`
- `RECO1C_EN`

Output:

- `LSa_EN`

Behavior:

- `LSa_EN` is asserted when at least two of the three inputs are
  asserted.

## 8.2 Stage 2 vote

Inputs:

- `RECO2A_EN`
- `RECO2B_EN`
- `RECO2C_EN`

Output:

- `LSb_EN`

Behavior:

- `LSb_EN` is asserted when at least two of the three inputs are
  asserted.

## 9. Recovery driver subsystem

There are five nominally identical recovery driver blocks:

- driver A
- driver B
- driver C
- driver D
- driver E

Each driver block contains:

- one dual high-side smart switch for the `PWR` outputs
- two independent low-side constant-current sink channels for the `RTN`
  outputs
- one local 24 V to 12 V linear rail
- digital fault outputs
- analog monitoring outputs
- hardware safety gating from RBF and E-stop

## 9.1 Per-driver exported signals

Each driver exports:

### Outputs to the load harness

- `RECO1_PWR_<X>`
- `RECO2_PWR_<X>`
- `RECO1<X>_RTN`
- `RECO2<X>_RTN`

### Control inputs

- `RECO1_EN`
- `RECO2_EN`
- `LATCH`
- `SEL`
- `ESTOP_DISABLE`
- `RBFTAG_DISABLE`

### Telemetry / fault outputs

- `FLT1`
- `FLT2`
- `SNS`
- `VREF_FB1`
- `VREF_FB2`
- local `+12V0` rail indication

## 9.2 High-side path behavior

The high-side path is built around `TPS272C45D`.

Nominal behavior:

- input rail: `P24V0`
- outputs: `RECO1_PWR`, `RECO2_PWR`
- enables: `EN1`, `EN2`, derived from the driver's `RECO1_EN` and
  `RECO2_EN` paths after hardware safety gating
- digital fault outputs: `FLT1`, `FLT2`
- analog sense output: `SNS`
- channel selector: `SEL`
- latch input: `LATCH`

Schematic notes document the intended behavior as:

- current-sense gain: approximately **1 V/A**
- current-sense bandwidth: approximately **1.6 kHz**
- high-side current limit: approximately **2.05 A**
- latch retry occurs on the **falling edge** of `LATCH`

`SNS` is a single analog sense output from the high-side device. The
`SEL` input selects which internal channel is presented on `SNS`.

## 9.3 Low-side return path behavior

Each driver contains two independent low-side constant-current sink
channels, one for `RECO1_RTN` and one for `RECO2_RTN`.

Each channel is built from:

- a `TLV9351` op-amp
- an `IPD110N12N3G` NMOS device
- local shunt and compensation components
- a filtered reference / feedback node exported as `VREF_FB1` or
  `VREF_FB2`

Nominal behavior documented in the schematic:

- sink current target: approximately **1.3 A** per channel
- reference node nominal value: approximately **0.784 V**
- feedback bandwidth: approximately **1.6 kHz**

Interpretation:

- when a channel is enabled, the corresponding `RTN` pin behaves as a
  controlled current sink
- external current flows only if the matching `PWR` output is also driven
  and an external load is present between the matching `PWR` and `RTN`
  pins

## 9.4 Safety gating

Both the high-side and low-side actuation paths are hardware-gated by:

- `RBFTAG_DISABLE`
- `ESTOP_DISABLE`

Nominal behavior:

- if either safety-disable signal is asserted, the driver blocks the
  enable/reference path needed for firing
- this inhibit action is local hardware behavior and does not depend on
  MCU software state once the signal is asserted

## 9.5 Local 12 V rail per driver

Each driver derives a private `+12V_RECO_<X>` rail from `P24V0` using an
`LT3065` regulator.

This rail powers:

- the smart-switch diagnostics
- the op-amp/sink circuitry
- local analog support components

This rail is not separately software-controlled.

## 9.6 Driver-to-stage mapping

| Driver | Stage group | Output labels | Analog owner |
|---|---|---|---|
| A | Stage 1 | `1A`, `2A` | MCU A |
| B | Stage 1 | `1B`, `2B` | MCU B |
| C | Stage 1 | `1C`, `2C` | MCU C |
| D | Stage 2 | `1D`, `2D` | MCU B |
| E | Stage 2 | `1E`, `2E` | MCU C |

## 9.7 Nominal firing behavior by stage

### 9.7.1 Stage 1

`LSa_EN` feeds both channel-enable inputs of drivers A, B, and C.

Therefore, when stage 1 is voted active and the safety paths are not
asserted:

- driver A enables `1A` and `2A`
- driver B enables `1B` and `2B`
- driver C enables `1C` and `2C`

### 9.7.2 Stage 2

`LSb_EN` feeds both channel-enable inputs of drivers D and E.

Therefore, when stage 2 is voted active and the safety paths are not
asserted:

- driver D enables `1D` and `2D`
- driver E enables `1E` and `2E`

## 10. Behavioral model summary for simulation

This section describes the intended logical behavior to model. It is not
an implementation prescription.

## 10.1 Power model

For a logical simulator, the nominal power subsystem should be modeled as:

- two independent external 48 V sources
- two independent always-on 48 V to 24 V converters with input-valid
  threshold behavior
- idealized active OR behavior into `P24V0`
- two independent always-on 24 V to 3.3 V converters
- idealized active OR behavior into `P3V3`
- ADC-visible scaled monitor rails:
  - `24V-VSNS`
  - `3V3-VSNS`

## 10.2 MCU / flight-computer model

The board should be modeled as exposing three SPI-addressable MCU
endpoints to the flight computer:

- shared `SCLK`
- shared `MOSI`
- shared `MISO`
- separate chip-select per MCU

Each MCU instance should be modeled with:

- its own local sensor bus and dedicated sensor set
- its vote output per stage
- access to all digital fault lines
- open-drain access to the shared latch nets
- its instance-specific analog ownership (`SNS`, `SEL`, `VREF_FB*`)

## 10.3 Voting-logic model

Both stage voting blocks should be modeled as pure combinational majority
logic:

- output high if at least two inputs are high
- output low otherwise

## 10.4 Recovery-driver model

Each driver block should be modeled as a dual-channel actuation device
with:

- two high-side switched `PWR` outputs from `P24V0`
- two low-side controlled-current `RTN` outputs
- local safety mask from `RBFTAG_DISABLE` and `ESTOP_DISABLE`
- two digital fault outputs
- one multiplexed analog `SNS` output selected by `SEL`
- two analog `VREF_FB` outputs representing each return channel's
  reference/feedback node

The simulator does not need transistor-level or regulator-level detail.
It should preserve the externally visible behavior described in this
section.

## 11. Out of scope

The following are intentionally out of scope for this document:

- full electrical verification of converter compensation networks
- thermal analysis
- PCB layout and grounding detail beyond what is software-visible
- optional / fallback population variants
- the legacy `VBAT-MID` path
- software algorithms or flight-state logic

## 12. Nominal software-facing conclusions

From firmware and logical simulation perspectives, the board should be
understood as:

- a **triple-MCU** recovery controller with one local sensor suite per
  MCU
- a **shared external SPI interface** from the flight computer into those
  three MCUs
- a **2-of-3 voted enable architecture** for each recovery stage
- a set of **five dual-channel recovery drivers** with hardware safety
  inhibition, analog telemetry, and digital fault reporting
- a **redundant power architecture** whose only software-visible power
  signals are the presence and scaled telemetry of `P24V0` and `P3V3`
